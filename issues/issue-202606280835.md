# 課題解決プロンプト: ai-env の非 --resume モードで provider/model が反映されない + --bash で provider/model 指定を可能にする

## 1. 役割定義 (Persona)
あなたは高度なソフトウェアエンジニアおよび問題解決のエキスパートです。以下のコンテキストとゴールを基に、正確かつ効率的な修正/実装を行ってください。

## 2. 背景・前提条件 (Context)

### 現状発生している問題

`ai-env` には現状 `--bash` / `--resume` の 2 つのオプションがあり、`buildInitScript` (`pi-projects.ts`) は以下 3 ブランチに分かれている:

1. `--bash` 指定時: `exec /bin/bash` のみ。**provider / model を指定する手段が一切ない**。
2. `--resume` 指定時: `pi-resume` シェル関数を経由してプロジェクト単位の `provider` / `model` / `apiKeyEnv` を組み立てる (ここは正しく動作している)。
3. `--resume` も `--bash` も未指定時: `defaultProvider` / `defaultModel` (プロフィール側のデフォルト) だけで `pi` を起動する。**プロジェクト側 (`projects["foo"].provider` 等) は完全に無視される**。

結果として、ユーザーは次のような `pi-projects.json` を用意しても、`ai-env` (resume なし) を実行したコンテナ内では `defaultProvider` / `defaultModel` (未設定なら空) で `pi` が立ち上がってしまう:

```json
{
  "projects": {
    "pi": {
      "session": "019f0b62-a4db-75ad-af9f-d78d43604605",
      "provider": "opencode-go",
      "model": "deepseek-v4-flash:xhigh"
    }
  }
}
```

加えて、`ai-env --bash` でコンテナに入って直接 `pi` を起動したい場合に「プロジェクト側の provider / model を引き継ぐ」「その場で任意の provider / model を指定する」どちらもできないため、ユーザーは `pi-resume` 経由でしか provider / model を渡せない。

### 再現手順

1. 上記のような `~/.config/ai-env/pi-projects.json` を配置する
2. プロファイル名を含むディレクトリで `ai-env` (resume なし) を実行
3. コンテナ内で `pi` が立ち上がるが、`pi-projects.json` の `projects["pi"].provider` (=`opencode-go`) と `model` (=`deepseek-v4-flash:xhigh`) は反映されず、プロフィールの `defaultProvider` / `defaultModel` (未設定なら未指定) で起動する

### 関連ファイル/コード

- `index.ts`
  - `program` の `option(...)` 定義 (`--bash`, `--resume` のみ。`--provider` / `--model` 未定義)
  - `RunContext` 型 (現状 `bashMode`, `resume`, credentials などを保持)
  - `runContainerCommand` → `buildInitScript` 呼び出し
- `pi-projects.ts`
  - `buildInitScript` (3 ブランチ: `bashMode` / `resume` / デフォルト)
  - `generatePiResumeFunc` (プロジェクト単位の provider / model / apiKeyEnv を `pi-resume` シェル関数として展開)
  - `buildOptionalFlag` ヘルパー
- `pi-projects.test.ts`: 既存テスト。修正時に壊さないこと。

## 3. 解決すべきゴール (Goal & Objective)

- [ ] `ai-env` (`--resume` / `--bash` なし) 実行時、カレントディレクトリのプロファイル名 → プロジェクト解決に基づき、プロジェクト側の `provider` / `model` / `apiKeyEnv` が `pi` に渡る(現状の `defaultProvider` / `defaultModel` へのフォールバックは維持)
- [ ] `ai-env --bash` 実行時、ユーザーが `--provider <p> --model <m> --api-key-env <env>` のような CLI オプションで provider / model / apiKeyEnv を指定でき、コンテナ内の bash 環境変数 `PI_PROVIDER` / `PI_MODEL` / `PI_API_KEY_ENV` (仮称) として渡る(プロファイル / プロジェクトのデフォルトも考慮したフォールバック)
- [ ] `--bash` モードでも、ユーザーの明示的指定がなければプロフィール / プロジェクトのデフォルト (現状の `--bash` と同じく追加動作なし) が適用される
- [ ] `pi-resume` 経由の挙動は壊さない
- [ ] `npx tsc` / `npx oxlint` / `npx tsx --check` / `npx vitest run` がすべてパスする
- [ ] `pi-projects.test.ts` に、上記 2 つの挙動 (非 --resume での project 反映 / --bash での CLI オプション受理) をカバーする新規ケースを追加

### 制約事項・注意点

- CLI オプション名は `--provider` / `--model` / `--api-key-env` など、既存プロジェクト設定のキー名と揃えてユーザーが推測しやすくする
- シェル関数 `pi-resume` の責務 (プロジェクト単位の組み立て) は変えず、**「`ai-env` デフォルト起動」でも同じ解決ロジック**を通す形に揃える
- `--bash` モードで `pi-resume` のようなシェル関数を bashrc に注入するかどうかは要判断 (`--bash` は「pi を起動しない」が目的なので、注入しない方が一貫性は高い。注入する場合はコマンドとして `pi-resume` のみ実行可能にする)。**デフォルトは注入しない**を推奨
- プロジェクト解決 (`detectProfileName` の派生: cwd からプロジェクト名を得る) は現状の `pi-resume` 関数の case 文と同じロジックを使う(例: ホスト側 `hostProjectName` をコンテナ環境変数として渡し、bash 内で `$HOST_PROJECT_NAME` として参照)。`ai-env` デフォルト起動では「`HOST_PROJECT_NAME` で `projects` を引き、provider / model / apiKeyEnv を抽出して `pi` に渡す」だけで十分
- `commander` のオプション引数にはバリデーション (`SAFE_SHELL_PATTERN` / `SAFE_MODEL_PATTERN` / `SAFE_ENV_NAME_PATTERN`) を必ず通す。`pi-projects.ts` 内の `requireSafeId` を再利用できるよう、必要なら export する
- README の `### 設定ファイルの構造` / `### プロファイルの自動判別` セクションの記述と整合させる

### 実装のヒント

`index.ts` 側:

```typescript
program
  .option("--bash", "pi を起動せずに bash シェルのみを起動する")
  .option("--resume", "pi-projects.json のセッションを引き継いで起動する")
  .option("--provider <provider>", "pi の provider を指定する")
  .option("--model <model>", "pi の model を指定する (model:thinkingLevel 形式可)")
  .option("--api-key-env <envName>", "pi の --api-key で参照する環境変数名");
```

`RunContext` に `provider?` / `model?` / `apiKeyEnv?` を追加。

`pi-projects.ts` の `buildInitScript`:

- `bashMode` ブランチで `PI_PROVIDER` / `PI_MODEL` / `PI_API_KEY_ENV` を `export` する(または bashrc に追記)
- デフォルト起動ブランチを `pi-resume` と同等のロジック (ホストプロジェクト名 → プロジェクト解決 → provider / model / apiKeyEnv 組み立て) に書き換える。`generatePiResumeFunc` の case 本体ロジックを共通関数 `resolveProjectOverrides(projects, hostProjectName, defaultProvider, defaultModel, defaultApiKeyEnv)` として抽出し、`pi-resume` の case と `buildInitScript` のデフォルトブランチの両方から呼ぶ形が望ましい
- `--resume` ブランチは変更不要
