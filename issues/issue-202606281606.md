# 決定記録: E2Eテストは導入しない

## 1. 役割定義 (Persona)
このドキュメントは方針決定の記録です。実装タスクではありません。将来の開発者が E2E テスト追加を検討する際に参照する意思決定ログとして残します。

## 2. 背景・前提条件 (Context)
- **検討の動機**: コードレビューで「テストが足りない」と指摘されたことを受け、E2Eテスト導入の是非を検討した。
- **このツールの性質**:
  - macOS 専用 CLI（`security` コマンド / `ipconfig getifaddr en0` / `container` ランタイムに依存）
  - macOS Keychain からクレデンシャルを取得
  - `gh auth token` で GitHub トークンを取得
  - Apple Container（`container` コマンド）でサンドボックス環境を起動
- **CI 環境での再現性**: GitHub Actions 等の汎用 CI では macOS 環境を保証できない。

## 3. 結論

**E2Eテストは導入しない**。理由：

| 必要なもの | 現実性 |
|---|---|
| macOS 環境 | GitHub Actions でも利用可能だが、self-hosted runner または macOS runner（有料）が必要 |
| macOS Keychain エントリ | テストごとに登録・削除が必要で、CI 上でセキュアに扱うのが困難 |
| `gh auth token` の偽装 | モック化しても本物の gh バイナリが必要になり、テスト並列化が困難 |
| Apple `container` ランタイム | Apple 純正のため CI で再現不可（macOS runner でも `container` パッケージは標準インストールされない） |
| 実際のコンテナ起動 | 起動時間のオーバーヘッド、イメージビルド前提（`Dockerfile` の `playwright install` 等で10分以上） |

## 4. 採用する代替戦略

- **手動動作確認**: 開発者のローカル macOS 環境で `npm run start` を実行して動作確認
- **ユニットテストの充実**: セキュリティクリティカルな `redactSecrets` / `buildVolumeArgs` を含む `index.ts` のテストを厚くする
- **統合テスト**: 必要に応じて `child_process` をモック化して「コンテナに渡される引数が正しいか」までを検証する（実コンテナ起動は不要）

## 5. この決定が覆る条件

以下がすべて満たされた場合、E2Eテスト導入を再検討する：

1. Apple Container が OSS 化され、Linux 上でも実行可能になる
2. macOS 上で GitHub Actions runner のセットアップ手順が標準化される
3. Keychain のテスト用仮想化が macOS 標準でサポートされる
4. プロジェクトのテスト対象が「個人用のローカルCLI」から「配布・CI前提のOSS」に変わる

それまでの間、この決定は変更しない。
