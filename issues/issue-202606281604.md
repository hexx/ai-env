# 課題解決プロンプト: 生成スクリプトの正規表現パースを bash -n 構文チェックに置換（テストの脆弱性解消）

## 1. 役割定義 (Persona)
あなたは高度なソフトウェアエンジニアおよび問題解決のエキスパートです。以下のコンテキストとゴールを基に、正確かつ効率的な修正/実装を行ってください。

## 2. 背景・前提条件 (Context)
- **現状発生している問題**: `pi-projects.test.ts` の以下のテストヘルパー関数が、生成されたシェルスクリプトを**正規表現でパース**して case 行を抽出しており、テストが生成スクリプトのフォーマットに密結合している：
  - `extractPiResumeCases`: 各行を `/^\s+[a-zA-Z0-9_-]+\) pi .*;;$/` でマッチ
  - `extractDefaultCaseBody`: `project="${HOST_PROJECT_NAME}"\ncase "$project" in\n([\s\S]*?)\nesac` でマッチ
  - `findDefaultCaseLine`: `extractDefaultCaseBody` の結果に対してさらに正規表現
  
  これらにより、スクリプト生成の空白・改行・コメント位置を変更しただけでテストが壊れるリスクがある。
- **再現手順**: `pi-projects.test.ts` 内の `extractPiResumeCases`、`extractDefaultCaseBody`、`findDefaultCaseLine` 関数の実装を確認する。
- **関連ファイル/コード**:
  - `pi-projects.test.ts`: 上記3関数
  - `pi-script.ts`: `buildInitScript`、`generateCaseBody`

## 3. 解決すべきゴール (Goal & Objective)
- [ ] 生成されたシェルスクリプトに対して `bash -n` を実行して**構文エラーがないこと**を確認するアサーションに切り替える
- [ ] 既存の正規表現パースによる構造検証は、よりロバストな方法（例: `awk` で case ブロックを切り出す、またはスクリプト生成ロジック自体にテスト可能な構造体を返すリファクタリングを併せて行う）に置換する
- [ ] これにより、フォーマット変更に対して耐性のあるテストになる
- [ ] 既存のテストケースの意味が維持されること

## 4. 期待する成果物 (Output Format)
- リファクタリング後の `pi-projects.test.ts`（または分割後の対応ファイル）の diff
- テスト実行結果
