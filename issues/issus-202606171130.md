# Incident Report: ホストの /home/pi/.bashrc が ai-env 実行で汚染

## 概要

2026-06-17 01:59 頃、`/home/pi/.bashrc` に `pi-resume` 関数が 16 行分追記される
汚染が発生した。`wc -l` が 113 → 129 に増加し、`grep -c "pi-resume"` が 0 → 1 になった。
別ターミナル(pts/0)でユーザが `ai-env` を実行した直後に発生。
incident 6 (issues/issus-202606132130.md で記録) と類似の汚染パターン。

## 発生状況

### 検出
- テスト中に `/home/pi/.bashrc` をチェックしたところ汚染を検出
- 直前のテスト(別ターミナル)では汚染なし(113 行・pi-resume 0 件)
- テスト中の `node dist/index.js --help / --version / --invalid-flag` では
  bashrc は変更されないため、テスト自体が汚染の原因ではない
- 同時に `pts/0` で以下のプロセスが動作中:
  ```
  /bin/bash -c cp -r /tmp/.ssh ~/.ssh && \
   chown -R $(id -u):$(id -g) ~/.ssh && \
   chmod 700 ~/.ssh && \
   find ~/.ssh -type f -exec chmod 600 {} \; && \
   mkdir -p ~/.config/herdr && \
   socat UNIX-LISTEN:/home/pi/.config/herdr/herdr.sock,fork,reuseaddr TCP:host.docker.internal:9123 & \
  cat << 'PI_RESUME_EOF' >> /home/pi/.bashrc \
  pi-resume() { ... } \
  PI_RESUME_EOF \
  exec /bin/bash
  ```

### 汚染内容
1 個の `pi-resume()` 関数が `/home/pi/.bashrc` 末尾(line 114〜)に追記された。
docker コンテナ内ではなくホストの `.bashrc` に直接書き込まれた。

### 復元
- 汚染前のファイル(`/home/pi/.bashrc.polluted-by-ai-env-YYYYMMDD-HHMMSS`)に
  バックアップ済み
- `head -113 /home/pi/.bashrc > /tmp/bashrc-clean && mv ...` で 113 行に復元
- 復元後: `pi-resume` 0 件・`ai-env` 0 件 ✓

## 根本原因

### 直接原因
docker コンテナが起動する init script の中で
`cat << 'PI_RESUME_EOF' >> /home/pi/.bashrc ... PI_RESUME_EOF` という
heredoc 構文を使っている。
本来このコマンドは docker コンテナ内で実行されるべきだが、
`/home/pi/.bashrc` という **絶対パス** がコンテナ内とホストで衝突する。

### 構造的問題
1. **絶対パス `/home/pi/.bashrc`**: docker run の --volume で host の
   `/home/pi` を mount していないのに、init script 内で `/home/pi/.bashrc`
   を直接書いている。docker コンテナ内で `pi` ユーザが `~/.bashrc` を
   触るのと同じパスに host のファイルが存在するため、host 側で
   bash が誤って解釈すると host 側に書き込まれる。

2. **heredoc のデリミタが bash シェルに渡る**: `PI_RESUME_EOF` は
   ユニークだが、コマンド全体が `bash -c '...'` で host 側に渡る
   可能性がある(spawnSync のエスケープが不完全な場合)

3. **テスト中の予期しない ai-env 実行**: テストと並行して
   別ターミナル(pts/0)で `ai-env` が実行されており、それが
   汚染の原因。テスト自体は汚染していない。

## 再発防止策(推奨)

### 短期(即時対応)
- テスト中は `ai-env` の実行を停止
- テスト前後で `/home/pi/.bashrc` の diff を確認
- incident 6 同様の手順(backup → head -N 復元)を incident playbook に登録

### 中期(code 側修正)
`buildInitScript` の `--init-script` で渡す heredoc のターゲットを
host と衝突しないパスに変更する:

**案 A: `/etc/profile.d/pi-resume.sh`**
- システム全体向け、root 所有
- host の `/etc` は docker コンテナ内のみなので衝突しない
- /etc/profile.d/ は全ログインシェルで自動 source される

**案 B: `/root/.bashrc`**
- docker コンテナ内では root ユーザで init script が実行される
  ことがほとんど(USER 設定による)
- host の `/root/.bashrc` を docker で mount しない限り安全

**案 C: docker 専用の一時ファイル**
- `--init-script` ではなく、`docker cp` で専用ファイルを送り込む
- 削除も `docker exec rm` で行う

### 推奨: 案 A `/etc/profile.d/pi-resume.sh`
最もシンプルで安全。host 側とパスが衝突しない。
