# 課題解決プロンプト: profiles にデフォルトの provider と model を設定できる機能を実装する

## 1. 役割定義 (Persona)
あなたは高度なソフトウェアエンジニアおよび問題解決のエキスパートです。以下のコンテキストとゴールを基に、正確かつ効率的な修正/実装を行ってください。

## 2. 背景・前提条件 (Context)
- **現状発生している問題**: 現在、`pi-projects.json` の `projects` セクションで各プロジェクトごとに `provider` と `model` を指定している。しかし、多くのプロジェクトで同じ `provider` と `model` を使う場合、各プロジェクトに重複して記述する必要がある。
- **期待される挙動**: `profiles` セクションに `provider` と `model` をオプションで指定できるようにし、プロジェクト側で未指定の場合はプロファイルのデフォルト値を使うようにする。
- **関連ファイル/コード**:
  - `pi-projects.ts`: `ProfileConfig` インターフェース、`parseProfileEntry` 関数、`generatePiResumeFunc` 関数
  - `index.ts`: `buildEnvArgs` 関数、`RunContext` 型
  - `pi-projects.example.json`: 設定例
  - `README.md`: ドキュメント

## 3. 解決すべきゴール (Goal & Objective)
- [ ] `ProfileConfig` インターフェースに `provider?: string` と `model?: string` を追加する
- [ ] `parseProfileEntry` 関数で `provider` と `model` を `SAFE_SHELL_PATTERN` でバリデーションして取り込む
- [ ] `generatePiResumeFunc` 関数で、プロジェクト側の `provider`/`model` が未指定の場合、プロファイルのデフォルト値を使うようにフォールバック処理を追加する
- [ ] `pi-projects.example.json` にプロファイル側の `provider`/`model` 設定例を追加する
- [ ] `README.md` にプロファイルのデフォルト値機能の説明を追加する
- [ ] 既存のテストがすべてパスすることを確認する
- [ ] `npx tsc` / `npx oxlint` / `npx tsx --check` がすべてパスすることを確認する

### 制約事項
- `provider` と `model` はオプションプロパティとし、既存の設定ファイルとの後方互換性を維持する
- プロジェクト側で `provider`/`model` が明示的に指定されている場合は、そちらを優先する（プロファイルのデフォルト値はフォールバックのみ）
- `SAFE_SHELL_PATTERN` (`/^[a-zA-Z0-9._-]+$/u`) でバリデーションする（`apiKeyEnv` と同じパターン）
- `apiKeyEnv` と同様の実装パターンに従う

### 実装のヒント
- `parseProfileEntry` 関数に `provider` と `model` のパース処理を追加（`apiKeyEnv` と同様のパターン）
- `generatePiResumeFunc` 関数でフォールバック処理を追加:
  ```typescript
  const provider = config.provider ?? profileProvider;
  const model = config.model ?? profileModel;
  ```
  - `profileProvider` / `profileModel` は `ProfileConfig` から取得
  - `generatePiResumeFunc` のシグネチャ変更が必要（`projects` だけでなく `profiles` も受け取る）
  - または `buildInitScript` のシグネチャ変更で対応
