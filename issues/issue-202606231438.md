# 課題解決プロンプト: ai-env コマンドで pi を終了させるとコンテナも終了するようにする

## 1. 役割定義 (Persona)
あなたは高度なソフトウェアエンジニアおよび問題解決のエキスパートです。以下のコンテキストとゴールを基に、正確かつ効率的な修正/実装を行ってください。

## 2. 背景・前提条件 (Context)
- **現状発生している問題**: 現在、`ai-env` コマンドで Docker コンテナを起動すると、コンテナ内で `pi` (pi-coding-agent) が起動する。しかし `pi` を終了させてもコンテナは終了せず、`exec /bin/bash` で起動された bash シェルが残り続ける。ユーザは `pi` 終了後にコンテナも自動的に終了することを期待している。
- **期待される挙動**: `pi` を終了させると、コンテナも自動的に終了する。
- **関連ファイル/コード**:
  - `pi-projects.ts`: `buildInitScript` 関数 (初期化スクリプト生成)
  - 現在の初期化スクリプトの末尾は `exec /bin/bash`

## 3. 解決すべきゴール (Goal & Objective)
- [ ] `buildInitScript` で生成する初期化スクリプトを修正し、`pi` 終了後にコンテナも終了するようにする
- [ ] `pi` が正常終了した場合と異常終了した場合の両方でコンテナが終了する
- [ ] 既存の `pi-resume` 関数や herdr-socat プロセスの動作に影響を与えない
- [ ] `npx tsc` / `npx oxlint` / `npx tsx --check` がすべてパスすることを確認する

### 制約事項
- `pi` は `pi-resume` 関数経由で起動される場合と、直接 `pi` コマンドで起動される場合の両方に対応する必要がある
- `pm2` で管理されている `herdr-socat` プロセスは、コンテナ終了時に適切にクリーンアップされる必要がある
- `exec /bin/bash` を削除する場合は、`pi` 起動前にユーザがシェル操作できる手段を残すかどうかを検討する

### 実装のヒント
- **アプローチ A**: `exec /bin/bash` を削除し、初期化スクリプトの最後で `pi` をフォアグラウンド実行し、`pi` 終了時に `exit` する
  ```bash
  # 現在の構造:
  exec /bin/bash
  
  # 変更後のイメージ:
  pi  # フォアグラウンドで pi を実行
  exit $?  # pi 終了時にコンテナも終了
  ```
- **アプローチ B**: `pi` を実行するラッパースクリプトを作成し、`pi` 終了時にコンテナを終了する
  ```bash
  # pi を実行し、終了後にコンテナを終了
  pi
  pm2 kill  # pm2 プロセスをクリーンアップ
  exit 0
  ```
- **アプローチ C**: `pi-resume` 関数を修正し、`pi` 終了後に自動的にシェルも終了する
  - `pi-resume` 内で `pi` を実行した後に `exit` を追加

### 検討事項
- `exec /bin/bash` を残す場合: ユーザは `pi` 終了後にシェル操作できるが、コンテナは終了しない (現状維持)
- `exec /bin/bash` を削除する場合: `pi` 終了後にコンテナが終了するが、シェル操作はできない
- `pi` を起動する前にシェル操作できる猶予期間を設けるかどうか
